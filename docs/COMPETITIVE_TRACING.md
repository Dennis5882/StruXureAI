## 0. 왜 이 문서가 필요한가

**타깃 제품인 MIDAS Gen NX에 이미 DWG→구조모델 순정 기능이 존재한다.** (`Structure` 탭 → `Create CAD Tracing File` / `Tracing File Import`)
"DWG를 처음부터 MIDAS로 옮긴다"는 포지션 그대로면 **순정 기능과 정면 중복**이다.
이 문서는 순정이 실제로 무엇을 하는지 정확히 규명하고, 우리가 어디서 이기고 어디서 지는지를 **같은 파일 실측 수치**로 고정한다.

> 결정적으로 유리한 점: 시연 영상이 **우리 표준 검증 파일 `B1F 1.dwg`** 를 쓴다. 1:1 대조가 가능하다.

## 1. 순정 Tracing 워크플로 (영상에서 관측)

| # | 단계 | 내용 |
|---|---|---|
| 1 | 진입 | `Structure` 탭 → `Create CAD Tracing File` / `Tracing File Import` |
| 2 | 내장 CAD | **Gen NX Drawing** 에디터가 열리고 `B1F 1.dwg` 로드 |
| 3 | **레이어 배정** | `Tracing/Grid Editor` — 레이어를 **수동으로** 각 역할에 끌어다 놓음<br>Center line←`CEN` / Column line←`CON_COLUMN` / Wall line←`CON_WALL` / Dimension←`A-DIM` / Grid Bubble←`A-AXN` / Undefined←`U_wall_masonry` |
| 4 | 그리드 | Center Line Info 표에서 X/Y축 평행선 목록 + 버블 스타일(`X1~X3`, `Y1~Y8`)을 **수동 확인·지정** |
| 5 | 영역 | `Define Region` 으로 범위 지정 → `Save` / `Export to Gen NX` |
| 6 | 임포트 | `Import DXF File` — **Scale Factor `0.001` 수동 입력** |
| 7 | 트리 | `CAD Tracing` 노드 생성: Center 58 / Column 241 / Wall 200 / Dimension 90 / Undefined 345 |
| 8 | 요소 생성 | `Create Elements with Tracing` → 기둥/벽 자동 생성 |

## 2. 순정 내부 동작 규명 (수치로 역산)

영상의 레이어 카운트를 우리 DXF 파서 결과와 대조하면 순정의 내부 처리가 드러난다.

### (a) 순정은 폴리라인을 선분으로 **분해(explode)** 한 뒤 처리한다

| 레이어 | 우리 파싱(원본 엔티티) | 분해 시 예상 | 순정 카운트 | 판정 |
|---|---|---|---|---|
| `CON_COLUMN` | LINE 41 + LWPOLYLINE 50 | 50×4 + 41 = **241** | **241** | ✅ **정확히 일치** |
| `CON_WALL` | LINE 177 + LWPOLYLINE 8 | 177 + (8개 분해분 23) = 200 | **200** | ✅ 일치 |
| `A-DIM` | LINE 90 (+LWPOLY 36, MTEXT 18) | LINE만 **90** | **90** | ✅ **정확히 일치**(문자·화살촉 제외) |
| `CEN` | LINE 56 | 56 | 58 | 근사(차 2) |

→ **순정은 "선분 집합" 수준에서 동작**하며, 치수 레이어에서 **문자를 버리고 선만 취한다.**
   이 마지막 사실이 결정적이다 → §4 (a).

### (b) 순정 요소는 **3D(층고 스팬)** 이며 벽은 4절점 PLATE다

```
기둥 60개 × 2절점(하단+상단) = 120  =  Nodes Created 120   ✅
벽  103개 × 4절점(PLATE)     = 412  =  160 created + 252 reused = 412   ✅
```

→ 기둥=2절점 수직 보요소, **벽=4절점 플레이트**. 절점 재사용률 61%로 인접 부재 절점 공유가 잘 된다.
→ 참고: **우리 `midasExport`도 벽을 수직 PLATE(4점)로 내보낸다.** 모델링 판단이 동일하다는 뜻이며, 우리 핸드오프 설계가 옳았다는 방증이다.

## 3. 동일 파일(`B1F 1.dwg`) 실측 1:1 대조

> StruXureAI v0.41.0, 크롭 없음, 두께 프로파일 raw. 재현: `?debug=1` + Playwright, `window.__store`.

| 항목 | 순정 Tracing | StruXureAI | 평가 |
|---|---|---|---|
| **기둥 수** | **60** | **60** | ✅ **완전 일치** |
| 기둥 단면 | (표시 없음) | 800×800 ×40, 800×2100 ×10, 2100×800 ×6, 1600×800 ×4 | 우리만 산출 |
| 기둥 그리드 참조 | 수동 버블 지정 | **60/60 자동** (`X2-Y1`…) | 우리 우위 |
| **벽 수** | **103** | **70** | ⚠️ §3.1 |
| 벽 총연장 | (미표시) | **398.8 m** | — |
| 벽 두께 | (미산출) | 200/300/350/425/475/500/550/800 실측 | **우리만 산출** |
| 그리드 축선 | 수동 확인 | **18개 자동**(X1~X10, Y1~Y8) | 우리 우위 |
| 축척 | **수동 `0.001`** | 자동 판정(B1F는 k=1로 정확히 거부) | **우리 우위** |
| 절점 | 3D 412 슬롯 | 평면 151 | 차원이 다름 |

### 3.1 벽 70 vs 103 — **누락이 아니라 분할 입도 차이**

총연장 398.8 m는 온전하다. 개수 차이는 **우리가 벽을 덜 쪼갠 것**이다.

```
우리 벽 70개
  + 그리드 축선이 내부를 관통하는 지점 36곳에서 분할  →  약 106개
순정 103개  ← 거의 정확히 설명됨
```

그리고 그 미분할은 **단 4개의 외곽 메가벽**에 집중돼 있다:

| 벽 | 길이 | 두께 | 관통 기둥 | 위치 |
|---|---|---|---|---|
| `W9` | **62,922 mm** | 433 | **8개** (X2-Y1 … X9-Y1) | 하부 외벽 전체 |
| `W43` | 46,950 mm | 500 | 6개 (X10-Y2 … Y7) | 우측 외벽 전체 |
| `W10` | 44,792 mm | 500 | 6개 (X1-Y2 … Y7) | 좌측 외벽 전체 |
| `W1` | 20,906 mm | 350 | 2개 (X6-Y5, X7-Y5) | 내부벽 |

→ **기둥 관통 22건.** 기둥 위치에서 분할하면 70 → 92개.

**이건 미관 문제가 아니라 MIDAS 핸드오프 결함이다.**
62.9 m 벽을 단일 PLATE로 내보내면 층고 3 m 기준 **종횡비 21:1** 이고,
그 벽이 지나가는 **기둥 8개와 절점을 공유하지 못한다** → 구조적으로 붙어있지 않은 모델이 된다.
순정은 이 지점들을 자동 분할해 절점을 공유시킨다(재사용률 61%가 그 증거).

> 이 결함은 순정과 대조하기 전까지 드러나지 않았다. 우리 자체 지표(벽 커버리지 96%)는 **총연장 기준**이라 분할 입도를 보지 못한다.

### 3.2 기타 관측된 우리 측 약점

- **자유단 59개** (검토 패널 기준). 순정은 절점 재사용 61%로 접합이 더 조밀하다.
- **비표준 두께 433, 467 mm** 잔존 → 면쌍 매칭 실패 잔여(대부분은 정상 스냅).

## 3.5 순정의 그리드·요소생성 내부 (영상 정밀 관측)

### (a) Center line — 순정은 "선 목록"을 보여주고 **사람이 검수**한다

`Tracing/Grid Editor` 우측 `Center line Info.` 표(레이어 `CEN` 선택 시):

```
      Type    Bubble   Bubble Style
 1  ▶ Lines parallel to X-axis          ← 그룹 헤더
 2   LINE     Y1       B Style-1
 3   LINE     (빈칸)
 4   LINE     (빈칸)
 5   LINE     Y2       B Style-4        ← Y2 중복 ①
 6   LINE     Y2       B Style-5        ← Y2 중복 ②
 7~9 LINE     (빈칸)
10   LINE     Y3       B Style-1
11   LINE     (빈칸)
12   LINE     Y4       B Style-1
13~16 LINE    (빈칸)
17   LINE     Y5       B Style-1
18,19 LINE    (빈칸)
20   LINE     Y6       B Style-1
21~24 LINE    (빈칸)
25   LINE     Y7       B Style-2
26   LINE     (빈칸)
27   LINE     Y8       B Style-1
28  ▶ Lines parallel to Y-axis          ← 그룹 헤더
29   LINE     X1       B Style-3
...
```

관측 사실:

1. **`CEN` 56개 선을 전부 나열한다.** 실제 그리드축은 18개뿐인데 나머지 38개(보조 중심선)도 행으로 남고 **버블이 빈칸**이다. X축 평행 그룹만 봐도 26행 중 17행이 빈칸.
2. **`Y2`가 두 행에 중복**되고 버블 스타일마저 다르다(`B Style-4` / `B Style-5`). 순정은 이 모호성을 **해소하지 않고 그대로 노출**해 사용자 판단에 맡긴다.
3. 축 명명 규칙은 우리와 동일: *X축에 평행한 선* → `Y1..Y8`, *Y축에 평행한 선* → `X1..`. 우리 `SGridAxis.dir` 의미와 일치.
4. 순정도 `Grid Bubble` 레이어(`A-AXN`)로 **버블 자동 채움은 한다.** 다만 채우지 못한 빈칸과 중복은 사람 몫이다.

> **대조:** 우리는 같은 도면에서 **18축(X1~X10, Y1~Y8)을 빈칸·중복 없이 자동 확정**한다.
> 다만 근거를 공정히 밝히면, 이 도면에는 `LAY_FOR_NGEN_BUBBLE_DATA`(TEXT 18)·`LAY_FOR_NGEN_DIMENSION_DATA`(TEXT 18)라는
> **MIDAS nGen 전용 보조 레이어가 미리 들어 있다.** 즉 이 샘플은 순정 tracing용으로 정비된 도면이며,
> 18이라는 숫자는 그 18개 TEXT와 정확히 일치한다. **일반 실도면에서의 우위는 별도 검증이 필요하다.**

### (b) 요소 생성 — `Create Elements with Tracing`

`Node/Element` 탭의 생성 패널(벽 생성 시점):

| 항목 | 값 |
|---|---|
| Element Type | **Wall** |
| Sub Type | **● Membrane** ○ Plate |
| Wall Type | **● Plate base** ○ CRB |
| Wall ID | **Auto Inc.** — 생성 전 `1` → 생성 후 `104` |
| Material | `1: Girder` |
| Nodal Connectivity | **● By Story** — Start `1F FL` → End `2 FL` |
| (By Selection 전용, 비활성) | Intersect ☑Node ☑Elem. / ☐ Create Intersecting Nodes |

관측 사실:

1. **`Wall ID: 1 → 104` 가 벽 103개를 확증**한다. MIDAS의 *Wall ID*는 같은 물리적 벽에 속한 플레이트들을 묶는 설계용 식별자다 → **"103"은 플레이트 장수가 아니라 물리적 벽 개수**.
2. 벽은 **Membrane / Plate base** 서브타입이며 **층(1F FL→2F) 스팬**으로 생성된다 → §2(b)의 4절점 회계와 일치.
3. `Intersect (Node/Elem.)`·`Create Intersecting Nodes` 옵션이 **존재한다.** 단 `By Story` 모드에서는 비활성이고 `By Selection` 전용이다.
   → 순정에는 **교차점에서 절점을 만들어주는 명시적 장치가 제품 기능으로 있다.** 우리가 §3.1에서 빠뜨린 바로 그 능력이다.
4. 기둥 생성 시 Element Type은 **General beam**(2절점 수직 보), 재질 `1: Girder`.



### 순정이 **못 하는 것** = 우리의 존재 이유

| # | 순정의 한계 | StruXureAI |
|---|---|---|
| (a) | **축척을 사람이 입력** (`0.001` 고정). 치수 레이어에서 **문자를 버리므로**(§2a) `DIMLFAC=0.5` 같은 2× 도면을 **원리적으로 탐지할 수 없다** | **자동 축척 캘리브레이션**(v0.41.0). 치수 *문자*를 읽어 기하/참값 비율 역산. 중국 시공도(PKPM/YJK)에서 factor=2 conf=1.00 검출 |
| (b) | **레이어를 전부 수동 배정** | **자동 분류**(`classifyLayer`) — 한/영/중·번체, 철근·치수·주석 부정키워드 제외 + 수동 오버라이드 |
| (c) | **부재 단면을 산출하지 않음** (선만 따라감) | 면쌍 매칭 **두께 실측** + 표준 양자화 + **평법 집중표주 라벨로 보 춤(depth) 파싱** |
| (d) | MIDAS 라이선스·설치 필요 | **브라우저**에서 동작, 라이선스 불필요 |

> **(a)가 가장 큰 해자다.** 순정은 치수 문자를 읽지 않기 때문에 도면 축척 왜곡을 구조적으로 감지할 수 없다.
> 우리가 `1#2#结构图.dwg`에서 발견한 "기하가 참값의 2배" 문제를 순정에 넣으면 **모든 부재가 2배인 모델이 조용히 생성된다.**

### 순정이 **더 잘 하는 것** — 인정하고 가져올 것

1. **부재 분할·절점 공유** (§3.1) — 명백히 순정이 낫다. 우리가 고쳐야 한다.
2. **3D 층고 요소를 한 번에 생성** — 우리는 평면 모델 후 층 스택.
3. **제품 내 통합** — 별도 도구 왕복이 없다.

### 포지션 재정의 (권고)

> ❌ "MIDAS를 대체하는 DWG→구조모델"
> ✅ **"순정 Tracing이 놓치는 것을 앞단에서 해결하는 전처리기"**
> — 자동 레이어 분류 + **자동 축척 보정** + 정밀 단면(두께·춤) → 더 깨끗한 입력/모델을 Gen NX로

특히 **대만/중국·동남아 도면**(한자 레이어, PKPM/YJK 2× 축척 관례)에서 순정의 (a)(b)가 정확히 무너진다.
우리 목표 시장과 우리 강점이 겹친다.

### 5.1 벽 기둥분할 적용 결과 (v0.42.0)

`splitLinesAtColumns`에 `snapToCenter` 옵션을 추가하고 벽에 `false`로 적용했다.

> **왜 벽은 기둥 중심으로 스냅하면 안 되는가** — 보 중심선은 개념적으로 기둥 중심에서 만나지만,
> **벽 축선은 기둥 중심에서 벗어나 있는 것이 정상**이다(외벽이 그리드가 아닌 외면 기준으로 그려짐).
> B1F 실측 이격은 **최대 150mm**이고 절점 병합 허용치는 **1mm**다. 중심으로 스냅하면 직선 외벽에 150mm 킨크가 생긴다.
> → **축선 위 수직발**에서 분할해 축선을 보존한다. 이격이 0인 구간에서는 수직발 = 기둥 중심이므로 절점 공유도 자동 성립.

| 지표 | 이전 | 이후 |
|---|---|---|
| 벽 개수 | 70 | **92** |
| 최장 벽 | **62,922 mm** | **9,031 mm** |
| PLATE 종횡비(층고 3m) | 21:1 | **3:1** |
| 기둥 관통 | **22건** | **0건** |
